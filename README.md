## Exact Steps to Run This Sprint

1. **Read `docs/00_SETUP_MYSQL.md`** — create the `credit_risk_analytics` database (30 seconds, you already have MySQL from Project 1)

2. **Read `docs/01_DATASET_AND_RISK_CONTEXT.md`** — download Lending Club data from Kaggle, place the gz file in `data/raw/`, and understand the banking context of each column (worth actually reading — this is interview material)

3. **Set up Python environment:**
   ```bash
   cd project2_fintech_credit_risk
   python3 -m venv venv
   source venv/bin/activate
   pip install -r requirements.txt
   ```

4. **Configure credentials:**
   ```bash
   cp .env.example .env
   # edit .env — set your MySQL password, DB_NAME=credit_risk_analytics
   ```

5. **Run the pipeline:**
   ```bash
   python scripts/01_clean_and_eda.py
   ```

   This will take a few minutes (dataset is ~2.26M rows). You'll see:
   - Loading progress
   - Target engineering summary (default rate)
   - Cleaning summary
   - 9 charts popping up + saving to `charts/`
   - A Risk Findings Summary printed at the end
   - Data loaded into MySQL table `loans`
   - `data/cleaned_loans.csv` saved for later sprints

## What This Script Does (Summary)

| Step | Action |
|---|---|
| Load | Reads only 34 relevant columns (not all 151) from the raw CSV — faster and safer |
| Target | Engineers binary `default` column from `loan_status`, dropping unresolved statuses |
| Clean | Parses term/emp_length strings, caps DTI and income outliers, imputes nulls with risk-justified logic |
| EDA | 9 risk-focused charts: default by grade, purpose, home ownership, employment, interest rate, DTI, loan amount, correlation heatmap, geographic |
| Load | Cleaned data → MySQL `loans` table + `cleaned_loans.csv` |

## Key Risk Findings You Should See

1. **Default rate increases sharply with grade** (A ≈ 5-7%, G ≈ 30%+) — grade is Lending Club's own risk model and it works
2. **`small_business` and `renewable_energy` purposes have the highest default rates** — much riskier than `debt_consolidation` or `credit_card`
3. **DTI and interest rate distributions clearly separate** defaulters from non-defaulters — visible overlap but distinct shapes
4. **FICO score is one of the strongest inverse correlates** with default

## Data Leakage — Why It Matters (Interview Gold)

This sprint deliberately EXCLUDES columns like `total_pymnt`, `recoveries`,
and `last_pymnt_d` — these only exist because we already know the loan
defaulted. Including them would make a model that "predicts" default
using information from the future. If an interviewer asks "how did you
handle data leakage," this project has a real, defensible answer.
